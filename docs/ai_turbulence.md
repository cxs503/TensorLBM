# HPC + AI Demonstration: AI Turbulence Models in TensorLBM

This document describes the `tensorlbm.ai` sub-package and the matching
platform agent tools that implement the full

```
Agent → modelling / solving / post-processing
      → dataset extraction
      → SQLite persistence
      → AI turbulence-model training
      → AI-enhanced LBM (LES collision)
```

pipeline.

## Why?

LBM-LES (Smagorinsky / WALE / Vreman / Dynamic-Smagorinsky) is already
shipped in `tensorlbm.turbulence`.  However the closure is algebraic and
hand-tuned per flow regime.  The AI sub-package demonstrates the typical
machine-learning workflow used in modern HPC + AI research: a small
neural network is trained on data **generated by the HPC solver itself**
and is then *plugged back* into the same solver as a sub-grid closure.

For pedagogical reasons the regression target is the Smagorinsky eddy
viscosity ν_t — this gives a fast, verifiable, fully reproducible
baseline.  Swapping the label for a higher-fidelity reference (DNS or
filtered DNS) only requires changing
`extract_les_samples_2d`, the rest of the pipeline is unchanged.

## Components

| Module | Purpose |
| --- | --- |
| `tensorlbm.ai.dataset` | Extract `(S_xx, S_yy, S_xy) → ν_t` samples from LBM velocity fields; save/load `.pt`. |
| `tensorlbm.ai.database` | SQLite layer with `runs`, `datasets`, `models` tables (`LBMDatabase`). |
| `tensorlbm.ai.model` | `EddyViscosityMLP` – a tiny MLP with a Softplus head (ν_t ≥ 0). |
| `tensorlbm.ai.train` | Adam + MSE training loop, returns history and saves the model. |
| `tensorlbm.ai.inference` | `predict_nu_t_2d`, `predict_tau_eff_2d`, `collide_ai_les_bgk` (drop-in for `collide_smagorinsky_bgk`). |
| `tensorlbm.ai.pipeline` | `run_ai_les_pipeline` – end-to-end orchestrator. |

All AI symbols are re-exported from the top-level `tensorlbm` package.

## Quick start (Python API)

```python
from tensorlbm import run_ai_les_pipeline, TrainConfig

result = run_ai_les_pipeline(
    work_dir="/tmp/ai_demo",
    nx=64, ny=64,
    tau=0.8, c_s=0.1,
    data_steps=60, sample_every=10,
    val_steps=40,
    train_config=TrainConfig(epochs=30, batch_size=2048),
)
print(result.to_dict())
```

For DNS-driven data generation in the same closed loop, switch to:

```python
from tensorlbm import run_ai_dns_pipeline, TrainConfig

result = run_ai_dns_pipeline(
    work_dir="/tmp/ai_dns_demo",
    nx=64, ny=64, dns_scale=2,
    data_steps=60, sample_every=10, val_steps=40,
    train_config=TrainConfig(epochs=30, batch_size=2048),
)
```

The call:

1. Initialises a small periodic 2-D field with non-trivial low-wavenumber
   structure.
2. Runs `data_steps` steps to generate training snapshots:
   - `data_source="les"`: LES run with `collide_smagorinsky_bgk`.
   - `data_source="dns"` / `run_ai_dns_pipeline`: higher-resolution BGK
     reference run, then block-averaged to the training grid.
   Snapshots are sampled every `sample_every` steps.
3. Builds an `EddyViscosityDataset` from those snapshots and writes it
   to `dataset.pt`.
4. Records the run, dataset and model in the SQLite database at
   `<work_dir>/ai_pipeline.db`.
5. Trains an `EddyViscosityMLP`, saves the weights to `model.pt` and the
   architecture JSON to `model.pt.json`.
6. Runs a short validation LBM simulation using `collide_ai_les_bgk`
   alongside a plain BGK baseline and reports stability + mean eddy
   viscosity.

## Replacing the LES closure

`collide_ai_les_bgk` matches the signature of
`tensorlbm.turbulence.collide_smagorinsky_bgk` exactly, so any existing
LES runner can use the trained AI model as a drop-in:

```python
from tensorlbm import load_model, collide_ai_les_bgk, stream

model = load_model("/path/to/model.pt")
for _ in range(n_steps):
    f = collide_ai_les_bgk(f, tau=0.8, model=model)
    f = stream(f)
```

## Design notes

* **Stability** — the network output passes through `Softplus(β=10)`
  so ν_t ≥ 0 by construction; `predict_tau_eff_2d` also clamps
  `τ_eff ≥ 0.5 + 1e-3` to keep the BGK operator well-conditioned even
  if the network briefly under-predicts.
* **CPU-friendly** — defaults run in seconds on a laptop CPU and the
  unit tests run in ~3 s.  No `cuda`, no extra dependencies beyond
  `torch` + Python stdlib (`sqlite3`).
* **Database is optional** — pure-Python users can import the dataset /
  training / inference modules directly without ever touching SQLite.
  Conversely, agent users can browse all artefacts through
  `ai_list_models` without writing a line of Python.
* **Reproducibility** — every public function accepts a `seed`
  argument and persists its configuration / training history in JSON
  inside the SQLite tables, so the full pipeline can be replayed.

## Future extensions

The same scaffolding can be extended to:

* **Higher-fidelity labels** — replace the algebraic Smagorinsky target
  with filtered DNS or experimental data.
* **Different architectures** — graph- or CNN-based closures; only the
  `EddyViscosityMLP` class needs to change.
* **3-D** — the architecture is dimension-agnostic; provide a
  `strain_rate_tensor_3d` and matching `collide_ai_les_bgk3d` and the
  rest of the pipeline carries over.
* **Online learning** — the database schema already tracks the
  provenance of every model (which dataset, which run); adding a
  retraining loop is a small extension.
